The "Z-Index" Gutter Issue

The "Z-Index" Gutter Issue (often called the Iframe Trap) is a common headache in web development. When you use Split.js with an iframe, the mouse often "slips" inside the iframe while dragging. Because the iframe is a separate document, the parent window loses track of the mouse movement, causing the resize to stutter or stop.

To understand this more, we need to look at how browsers handle mouse events and how iframes act like "black holes" for data.

The Core Problem: Event Hijacking

When you drag a divider (the gutter) in Split.js, the browser tracks your mouse movement via mousemove events on the parent window.

As long as your cursor stays over the gutter or the parent container, everything is fine. However, if you move the mouse too fast, the cursor might drift away from the thin gutter and hover over the iframe next to it.

  • The "Black Hole" Effect: An iframe is essentially a separate window nested inside your page. Once the cursor enters the iframe's boundary, the parent window stops receiving mouse events.
  • The Result: The Split.js logic "freezes" because it no longer knows where your mouse is. The divider stops moving, or it "stutters" as it tries to catch up when the mouse happens to exit the iframe.

The Solution: CSS Pointer-Events

The simplest and most effective way to fix this is to use the CSS property pointer-events.

By setting pointer-events: none; on the iframe, we tell the browser to completely ignore mouse events on that element. This forces the browser to pass the mouse events through the iframe and up to the parent window, allowing Split.js to track the mouse movement continuously.

onDragStart (The Cloak)

app.js 
document.getElementById('preview-frame').style.pointerEvents = 'none';

When you start dragging, the CSS property pointer-events: none is applied to the iframe. This tells the browser: "Ignore this element." The iframe becomes "transparent" to the mouse. Even if your cursor is physically over the iframe, the mouse events "fall through" it and are captured by the parent window instead.

onDragEnd (The Return)

app.js 
document.getElementById('preview-frame').style.pointerEvents = 'auto';

When you stop dragging, the CSS property pointer-events: auto is applied to the iframe. This tells the browser: "Pay attention to this element again." The iframe becomes "solid" to the mouse. Even if your cursor is physically over the iframe, the mouse events "fall through" it and are captured by the parent window instead.

Why is it called a "Z-Index" Gutter Issue?

While the fix uses pointer-events, the problem is often categorized under "Z-Index" or layering because it deals with which "layer" is catching the user's input.

In some alternative fixes, developers place a transparent div overlay directly on top of the iframe (higher Z-Index) during the drag. This achieves the same goal: ensuring the parent window keeps "possession" of the mouse events.

Full Solution

Here is the full solution with the pointer-events fix to replace the 2 split() calls in app.js:

app.js 
// Split helper functions
const disableIframe = () => {
  document.getElementById('preview-frame').style.pointerEvents = 'none';
};
const enableIframe = () => {
  document.getElementById('preview-frame').style.pointerEvents = 'auto';
};

// Horizontal split: editor | preview+console
Split(['#editor-panel', '#right-panel'], {
  sizes: [50, 50],
  minSize: 200,
  gutterSize: 6,
  cursor: 'col-resize',
  onDragStart: disableIframe,
  onDragEnd: enableIframe
});

// Vertical split inside right panel: preview | console
Split(['#preview-panel', '#console-panel'], {
  direction: 'vertical',
  sizes: [70, 30],
  minSize: [100, 50],
  gutterSize: 6,
  cursor: 'row-resize',
  onDragStart: disableIframe,
  onDragEnd: enableIframe
});
